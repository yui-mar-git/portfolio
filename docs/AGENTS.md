# AIエージェント向け 編集・作業ルール

このファイルは、AIエージェントがこのポートフォリオプロジェクトを編集する際に必ず参照・遵守すべきルールをまとめたもの、命名規則、ガードレール、アローリストです。

## 1. プロジェクト構成の基本理解

### 1.1 サイト階層構造

サイトは以下のような階層構造を持っています。

```
■ ルートページ（トップページ）
 └── /   (ポートフォリオトップ)

■ 第二階層（セクションページ / 子ページ）
 ├── /portfolio/   (ゲーム一覧)
 ├── /profile/     (プロフィール)
 └── /contact/   (お問い合わせ)

■ 第三階層以降（個別ページ / 詳細ページ / 子孫ページ）
 ├── /portfolio/tower-defense/
 ├── /portfolio/roguelike/
 └── ...その他各個別ページ
```

**【階層の呼称ルール】**
- 自身の現在のページから見て、上の階層にあたるページ群を **「直系尊属」「親」** と呼ぶことがあります。
- 自身の現在のページから見て、下の階層にあたるページ群を **「直系卑属」「子」** と呼ぶことがあります。

### 1.2 ディレクトリの役割と編集対象
プロジェクトのコードやリソースを編集・追加する際、以下のディレクトリの役割を順守してください。
- **`src/`**: コーディングの主要な作業場所です。Astroファイル、CSS、JavaScriptなどの処理されるソースコードはすべてこの中に配置・編集します。
- **`public/`**: 画像や音声など、ビルド時にそのまま配信される静的アセットを配置します。Astroによって加工されないファイルが対象です。
- **`dist/` (または `.astro/`)**: これらはビルドされた最終出力結果が格納されるディレクトリです。ファイルを直接編集しても自動的に上書きされてしまうため、**絶対に編集しないでください**。

## 2. コーディング規約とスタイルルール

### 2.1 CSS・スタイリングに関するルール
- **基本ルール**: クラス名は、そのHTML要素の「役割」や「内容」が一目でわかるセマンティック（意味論的）な名称にすること。
- **禁止事項**: `class-1`, `box-a`, `wrap-2` のような連番や、意図が読み取れない無意味な命名は一切行わないこと。
- **推奨される命名**: `nav-button`, `article-title`, `user-profile-card` など、その要素がシステム内で何を果たしているかを示す英単語を使用すること。
- **`!important` の使用禁止**: CSSでスタイルを定義する際、無闇に `!important` を使用しないでください。どうしてもそれがないと成り立たない場合を除き、適切な詳細度（セレクタの強さ）を用いてスタイルを上書きしてください。他の要素の見た目に意図しない影響を与えるのを防ぐためです。
- **インラインスタイルの禁止（※例外あり）**: スタイルは基本的に class で実装してください。タグごとに `style` 属性を直接記述（インラインスタイル）すると、コードの記述量が増える上に、CSSを用いて共通のスタイルを他の場所へ横展開する際に邪魔になってしまいます。
  - **【例外】**: JavaScriptで動的に値を変化させる必要があるプロパティ（HPゲージの幅 `width` やキャラの座標 `top`, `left` など）については、インラインスタイルの使用を許可します。
- **アセットのパス指定**: 画像などのアセット（asset）を参照する際、外部URL（http~）の直書きは避け、ドメインルートからのパス（例: `/portfolio/tower-defense/asset/...`）または相対パスを使用してください。

### 2.2 コードの記述とコメントの付与
ユーザー自身の学習とコードレビューを容易にするため、AIエージェントが新たにコードを記述・変更する際は、可能な限り以下の配慮を行ってください。
- **意図の明記**: 追加したコマンドやロジック、あるいは適用したスタイルが「なぜ必要なのか」「どのような意図で動くのか」を、コード内のコメントアウト等で極力簡潔に残すこと。
- これにより、エージェントが作業を終えた後でも、ユーザー自身がコードを読んで実装の意図をすぐに理解・学習できるようにします。

## 3. ゲームページ固有のルール

### 3.1 コンポーネント設計と分離
- **ゲーム本体の分離実装**: ゲームのロジック（PhaserなどのJSファイル）だけでなく、ゲーム用UI（強化画面やリザルト画面など）もAstroコンポーネント等として可能な限り別ファイルに切り出し、ページファイルから呼び出す形で実装してください。これは、ゲーム本体にあてたスタイルが、ゲームページの「概要」や「学習テーマ」などのスタイルと衝突し、レイアウト崩れを引き起こすのを避けるためです。
- **スタイルとロジックの相互影響の防止**: ゲーム内ロジックやUIを修正する際は、サイト全体（ページ側）のデザインや挙動に影響を与えないようにしてください。逆にページ側を修正する際も、ゲームロジックやゲーム内UIに影響を与えないよう注意してください。これを防ぐため、ゲーム内で使用するCSSには特定のIDやクラス（例: `.game-container` など）を必ずスコープとして指定し、スタイルがゲーム内にのみ適用されるようにしてください。ページ側のスタイルについても同様に、ゲーム内に漏れ出さないよう適切なセレクタ設計を行ってください。

### 3.2 ページコンテンツと表示ルール
- **コンテンツテキストの取得元**: ゲームの「概要（description）」は必ず microCMS から呼び出して表示させてください。ただし、「学習テーマ」「使用技術」「使用素材」については管理の煩雑化を防ぐため、Astroファイル内にハードコードして構いません。
- **使用言語・ツールの並び順**: 「開発言語」 -> 「フレームワーク」 -> 「その他のツール」 の順に並べて表示してください。
- **使用素材の並び順**: 「画像・イラスト」 -> 「フォント」 -> 「SE・BGM」 -> 「アイコン」 の順に並べて表示してください。
- **未使用項目の追加禁止**: 「使用言語・ツール」および「使用素材」のいずれにおいても、そのゲームで実際に使用していない項目を勝手に追加しないでください。
- **レスポンシブ・テーマ対応**: レスポンシブ対応や、ライトモード・ダークモードの切り替えによって、ゲーム画面内のテキストの可読性や、レイアウトの変化によるゲーム性（プレイフィール）が変わらない・損なわれないように実装してください。

### 3.3 ゲーム間の干渉防止と共有ファイルの扱い
- **他ゲームへの影響の遮断**: 特定のゲーム（例：タワーディフェンス）のロジックやデザインを修正する際、他のゲーム（例：ローグライク）に影響を与えてはいけません。ゲーム固有のバグ修正や機能追加は、必ず各ゲーム専用のディレクトリ（`src/scripts/<ゲーム名>/` や `src/components/portfolio/<ゲーム名>UI.astro`）内で完結させてください。
- **共有ファイルの変更時の注意**: `global.css` や `GameLayout.astro`、共通アセットなどの「全ゲームで共有されているファイル」を特定のゲームの都合で安易に変更しないでください。やむを得ず共有ファイルを変更する場合は、変更後に必ず**すべてのゲーム**のページにアクセスして、動作・レイアウトが崩れていないか検証（回帰テスト）を行ってください。

## 4. クレジット管理ルール

ゲームおよびポートフォリオのクレジット表記においては、以下のルールに則り、動的なアセット追加に対応できる拡張性の高い表記を行ってください。

1. **【使用技術】**
   - 言語、エンジン、主要パッケージは名称の横に (ライセンス名) を明記すること。
   - 例: `Astro (MIT License)`
2. **【使用素材（アセット名）】**
   - 「素材カテゴリ」「製作者/サイト名」「ライセンスまたは規約準拠の明記」「配布元URL」を1セットとして記述すること。
   - 例: `画像・イラスト：いらすとや（利用規約に準拠）[URL]`
3. **【AIの利用に関する透明性】**
   - 成果物における生成AIアセット（画像・音声）の不使用と、コード補助としての利用範囲を明記すること。
   - 例: `Google Antigravity, Gemini, Claude`
